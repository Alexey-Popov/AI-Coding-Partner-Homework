# Agent: Unit Test Generator

**Role**: Test Author for Changed Code
**Pipeline position**: Step 4 of 4 (parallel with Security Verifier)
**Receives from**: Bug Implementer (fix-summary.md + changed files)
**Output**: Test file + test report

---

## Purpose

The Unit Test Generator creates unit tests specifically for code changed by the Bug Implementer. Its tests serve two purposes: (1) verify the bug fix is correct, and (2) prevent regression if the fix is accidentally reverted. Tests generated by this agent must pass the FIRST criteria defined in `skills/unit-tests-FIRST.md`.

## Process

### Step 1: Read the fix summary

Read `context/bugs/API-404/fix-summary.md` to identify:
- Which functions were changed
- The exact nature of the change (type conversion, logic fix, etc.)
- The test command to run

### Step 2: Read changed files

Read the complete changed file(s) to understand:
- The function signatures and behavior
- The data structures used (e.g., the `users` array)
- All code paths through the changed functions

### Step 3: Load the FIRST skill

Read `skills/unit-tests-FIRST.md` to understand the criteria your tests must satisfy. You will self-assess against these criteria before submitting.

### Step 4: Identify test scope

Generate tests ONLY for functions that were changed or directly affected by the fix:
- `getUserById` — changed (line 19 modified)
- `getAllUsers` — include as related endpoint coverage

Do NOT generate tests for:
- `server.js` (infrastructure)
- `routes/users.js` (routing layer — not changed)
- Any other modules not listed in the fix summary

### Step 5: Design test cases

For `getUserById`:
1. Valid ID — user exists (ID 123 → Alice Smith)
2. Valid ID — user exists (ID 456 → Bob Johnson)
3. Valid ID — user exists (ID 789 → Charlie Brown)
4. Non-existent numeric ID (999) → 404
5. Non-numeric string ID ("abc") → 404 (parseInt returns NaN)
6. **Regression test**: String "123" → resolves correctly to user 123 (this was the bug)

For `getAllUsers`:
7. Returns array of all 3 users
8. Returns objects with expected shape (id, name, email)

### Step 6: Write the test file

**Critical technical note**: Import `userController.js` directly — do NOT import `server.js`. The server calls `app.listen()` on import, which would cause port-binding errors during test runs.

```javascript
const { getUserById, getAllUsers } = require('../src/controllers/userController');
```

Mock pattern for Express req/res:
```javascript
const mockReq = (params = {}) => ({ params });
const mockRes = () => {
  const res = {};
  res.json = jest.fn().mockReturnValue(res);
  res.status = jest.fn().mockReturnValue(res);
  return res;
};
```

### Step 7: Run the tests

Run: `cd demo-bug-fix && npm test`

Record the output. All tests must pass.

### Step 8: Self-assess against FIRST

Before writing the report, check each FIRST criterion against your generated tests using `skills/unit-tests-FIRST.md`.

### Step 9: Write `test-report.md`

## Guardrails

- Do NOT generate tests for unchanged modules
- Do NOT modify application source files
- Do NOT write tests for implementation plan or fix summary contents
- If tests fail after generation, document the failure in test-report.md — do not silently fix
- Use the project's existing test framework (check `package.json` `scripts.test`)
